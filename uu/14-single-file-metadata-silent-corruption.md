# uu-14 — 全量上書きファイルに「骤減検知」がない：静默データ損失の死角

> 来源：プロジェクト管理ダッシュボードの更新作業中に発覚した構造的盲点

## 背景

プロジェクト一覧を単一の Markdown ファイル（`projects-index.md`）で管理していた。スキャンスクリプトが毎回全量上書きする設計で、ある操作後にメニューバーの表示が「115件 → 32件」に変わって初めて異常に気づいた。

---

## ① Unknown Unknown：全量上書き + 部分スキャン = 静默縮小

**実証**：  
スキャンスクリプトが macOS の TCC（プライバシー権限）でデスクトップフォルダを読めないまま走ると、アクセスできたフォルダ（`~/Projects/`）だけをスキャンして「正常終了」する。上書き後のファイルは 115件 → 32件に減っているが、エラーも警告も出ない。

**機構**：  
`script --save` の実装が「全量洗い替え」のため、スキャン漏れ = データ欠落が即座に起きる。「書き込み成功」と「内容の正確さ」は別の概念だが、スクリプトは前者しか検証しない。

**修法**：  
```bash
N=$(grep -c '^### ' ~/.claude/projects-index.md)
[ "$N" -lt 80 ] && echo "WARN: only $N projects, scan may be partial — aborting" && exit 1
```
閾値は履歴最低値から設定。「80件未満なら書き込まない」にすることで部分スキャンの上書きを防げる。

---

## ② Unknown Unknown：macOS TCC の権限は「プロセス親子」を問わない（バイナリ単位）

**実証**：  
「同じスクリプトを実行して」という指示が、Terminal では成功・Claude のシェルサブプロセスでは失敗・launchd + python3 では成功、と3パターンの結果になった。

**機構**：  
macOS TCC は実行可能バイナリ単位で Full Disk Access を管理する。Terminal.app 経由の bash と、他のアプリ経由で spawn された bash は別エンティティとして扱われる。「子プロセスが親の権限を継承する」という Unix の一般的な直感は TCC 保護対象のディレクトリでは成立しない。

**Vibe coding への影響**：  
エラーが「コードのバグ」なのか「権限の問題」なのか区別できない。同じコマンドが文脈によって成功/失敗するため、「動いたこともある」が「なぜ動かないかわからない」状態が生まれやすい。

**修法**：  
- Desktop を触る操作は launchd ジョブ（`program=/usr/bin/python3`、FDA 取得済み）に移譲する
- Claude bash で失敗したとき「TCC かもしれない」を一番最初の仮説に置く

---

## 最高杠杆：全量上書きファイルには必ず下限チェックを入れる

「書き込み成功」≠「内容が正しい」。件数や行数の下限チェックを出力直前に挟むことで、部分スキャン・途中終了・権限ブロックを「静默成功」から「検出可能な失敗」に変えられる。コスト：5行。リカバリコスト：git log を掘る数十分。

---

## 誠実な留保

- 今回は git snapshot があったため 115件を復元できた。git がなければ永久損失だった
- TCC の正確な継承ルールは Apple の公開ドキュメントに詳述されていない部分がある。「バイナリ単位」という説明は動作観察からの推論を含む
- 「骤減が起きたのは今回が初めてか？」は不明。頻度データがないため構造的問題か偶発的問題かの断定はできない
- 分析者自体が未検証の検証器である
