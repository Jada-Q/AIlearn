# uu-37 · 分发是反应式的：即便发，也是想起一个渠道发一个，没计划、没成功信号

> 来源：一场"给一件已完成的作品写社媒发布文案"的短会话中挖出。作品（一个数据可视化视频）两天前就做好了，这天才来要发布文案，而且先要 X 版、拿到后再要小红书版。
> 亲缘：这是 [uu-04（建好流水线却没按发布键）](./04-building-the-pipeline-instead-of-shipping.md) 和 [uu-35（生产反射穿了学习的戏服）](./35-production-reflex-disguised-as-learning.md) 的**下一层**。那两条说的是"根本不分发"；这条说的是——**当真的分发时，它依然是未成体系的**：反应式、一次一渠道、没有"算不算成功"的定义。

## 洞（最高杠杆）：分发的"手"早造好了，缺的是"脑"

已经有一整套分发工具（各平台的发布命令、录屏工具都齐了）。所以卡点**不在工具**——不在"怎么发"。卡点在：没有一个前置计划回答"这件作品投哪几个渠道 / 各渠道用什么框架 / 发出去后怎么知道有没有效"。

工具解决了"怎么发"，没解决"发什么、发到哪、发了算不算数"。于是分发退化成一连串反应式动作：想起 X 就发 X，想起小红书再补小红书。

### 实锤

- 作品文件两天前就产出（文件名甚至还带 `-draft`），两天后才来做分发。生产与分发之间有一道自然的、无人守的缝。
- 分发请求是**顺序反应式**的：先"发 X"，拿到成品，再"写小红书"。不是一次把渠道矩阵规划好，而是一个渠道一个渠道地临时想起。
- 全程没有出现"这条发出去，我用什么判断它有没有效"这类问题——**没有成功信号，就没有反馈回路**。

### 机制

生产完成会给人"进展感"，于是注意力自然停在"做下一个"。分发是另一种肌肉：它要求对**已存在**的东西收口，还要求容忍"发出去可能没人理"的暴露感。因为分发没被设计成"完成"的一部分，它就永远是事后补的、临时起意的、做完就忘的——最致命的是**没有度量**，所以哪怕发了十条，也学不到"哪种框架、哪个渠道真的有人看"。

关键错位：把分发当成"发出去就行"的一次性动作，而它其实是个**需要闭环的实验**（假设 → 发布 → 读信号 → 迭代框架）。没有信号那一环，前面发多少都不累积成能力。

## 立即可做的检查动作

1. 打开自己的代码托管主页，数两个数：**已完成的作品数** vs **其中真正发过任一社媒的数**。差距越大，越说明"生产 >> 分发"。
2. 从下一件作品起，把"完成"的定义里加一行**分发清单**：渠道 × 各渠道文案角度 × 一个"算成功"的信号（点赞/点击/评论数）。让分发和成功信号一起在 ship 那一刻就被规划，而不是两天后反应式地补。

## 诚实反束

单场观察，非判决。最硬的那句结论（"大量作品其实从没被分发"）是从项目清单**推断**出来的，本场只实锤到"这一件作品的分发是反应式、隔了两天、没提成功信号"。写下这条的分析者本身也是没被验证的验证器——若这些作品本就不为传播而做（纯表达 / 纯积累），那"分发滞后"根本不是问题，本条不成立。
