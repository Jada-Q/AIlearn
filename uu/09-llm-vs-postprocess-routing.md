# UU-09 — 结构型需求不该交给 LLM：生成 vs 后处理的选择盲点

> 2026-06-14 · 从一次 WebGL 网站生成器（多页路由功能）的三轮推翻里挖出来的

---

## 背景

在构建一个"对话式 WebGL 网站生成器"时，需要让生成的 HTML 包含 nav + 4个子页面 + hash 路由切换。前后改了3次 prompt 都不稳定，第四次改成在服务端后处理注入路由脚本，一次搞定。

---

## UU1：把"让 LLM 生成正确代码"当作默认路线

**实锤**：路由功能前后改了3次 prompt（骨架前置 → 内容指令 → display/opacity 语义修复），每次都在调整"给 LLM 的指令"，而不是第一轮就问——

> 「这个任务适合让 LLM 做吗？」

**机制**：LLM 生成代码时没有执行环境，它无法验证自己写的路由在 iframe+srcdoc 场景下是否跑通。更根本的是：**结构保证型需求的验收标准是 0/1（对或不对），而 LLM 的输出是概率分布**——两者天然不匹配。

**分类**：
- **内容型需求**（文案、样式风格、WebGL shader 效果）→ 适合 LLM，容忍模糊
- **结构型需求**（路由逻辑、导航激活状态、页面切换）→ 不适合 LLM，要代码写死或后处理注入

**修法**：开工前先问「内容型还是结构型？」。结构型需求默认走后处理——LLM 生成骨架内容，服务端注入确定性逻辑。这条若在第一轮就执行，省掉三轮推翻。

---

## UU2：模型选型缺少能力预测框架

**实锤**：项目里先后考虑了 Gemini API → 本地 Gemma 4（2B/4B）→ 最终回 Claude CLI。切换本地模型时没有提前测试，上线后发现8项结构检查全 NO（nav/路由/子页面全缺失），只能切回。

**机制**：参数量不是能力的唯一指标，但粗略规律存在：
- <7B 模型很难稳定遵循"生成含路由逻辑的多页HTML"这类需要长上下文结构指令跟随的任务
- 对话优化模型 ≠ 代码生成模型（Gemma 4 e2b/e4b 属于前者）
- 指令遵循能力（instruction following）和代码生成能力是两个独立维度

**修法**：新模型替换主路径前，先30秒能力测试：

```bash
curl .../v1/chat/completions -d '{"messages":[{"role":"user","content":"Output a minimal HTML page with nav HOME WORK ABOUT. Only HTML."}]}'
```

返回对了再切，不对就保留原路径。

---

## 最高杠杆

**「结构保证型需求默认后处理，不走 prompt 工程」**

实际执行形式：在生成 API 的 route 里，LLM 输出 HTML 之后，用正则/字符串操作注入确定性的路由脚本，并移除 LLM 可能生成的冲突版本。效果：LLM 可以生成任意路由实现，都会被替换成我们的版本，功能100%稳定。

---

**诚实反束**：以上基于单场会话行为观察，不是对能力的判决。「结构型/内容型」的分界在真实工程里也不总是清晰——这是一个启发式框架，不是铁律。分析者本身也是没被验证的验证器。
