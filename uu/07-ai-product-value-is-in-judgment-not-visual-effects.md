# uu-07 — AI 产品的差异化在判断力，不在视觉效果（+ 功能难度与用户感知价值是两个独立维度 / 约束先于需求 / 所有功能都有质量基准线）

> 这四条 unknown-unknowns 从一场 5 小时黑客松里浮出来。任务是构建 AI 妆容顾问 app，在追求"效果图"功能的过程中，连续三次技术方案翻车，逼出了四条本来可以更早知道的认知盲点。

## 背景

构建一个 AI 妆容顾问：用户上传自拍 → 选场景（上班/约会/派对）→ AI 诊断 → 显示 BEFORE（标注问题）和 AFTER（效果图）。其中 AI 诊断（Gemini 2.5 Flash Vision 读脸）很快做好了。然后花了大量时间追 AFTER 效果图：Canvas 几何色块 → Gemini 图生图（API 不支持图像编辑）→ MediaPipe AR 妆容叠加。三种方案都做了，没一个达到期望质量。

最终结论：不做效果图，整个产品印象反而更好。

## UU-1（最高杠杆）：质量基准线 — 低于基准 = 负资产

- **实锤**：Canvas 色块效果图、MediaPipe AR 妆容都在用户预期基准线以下。用户的潜在参照是 Snapchat / Instagram 滤镜级别。完全不做效果图、只展示 AI 诊断 + 色卡建议 + 步骤，整体产品印象反而更好。
- **机制**：功能质量有两个区间——高于用户期望基准线 → 加分；低于基准线 → **负资产**（破坏用户对整个产品的信任，而不是中性地"没做"）。不同功能类型的基准线高低不同：AR 视觉效果基准极高（Snapchat level），AI 文字诊断基准相对低。
- **为什么是 unknown unknown**：直觉上认为"做了总比没做好"，但低于基准的功能让用户做负面对比——"这比手机自带滤镜还差"——这种印象会传染给产品里其他本来不错的部分。
- **修法**：做任何视觉/体验类功能前，先估"这类功能用户的期望基准在哪里"，再问"我们能达到那条线吗"。达不到 → 不做，把资源放到基准线更低的功能上。具体检查动作：对每个功能标「用户期望基准（低/中/高）× 我们能达到的质量（低/中/高）」，只做"能达到 ≥ 基准"的功能。

## UU-2：AI 产品的竞争力在判断力，不在视觉效果

- **实锤**：追效果图的时间远超打磨 AI 诊断质量的时间。但 AI 诊断（读懂这张脸的具体问题、给出有说服力的色调建议）才是没人能复制的差异化——视觉效果只是包装，且有充分竞品存在（Snapchat/Instagram）。
- **机制**：视觉效果"可感知"（immediately impressive）但未必稀缺；AI 判断力"有价值"（actually useful）但不够炫。人类直觉优化可感知的，但 AI 产品的护城河在判断力。一个 AI app 如果判断力和通用搜索/工具没区别，视觉做得再好也没有留存理由。
- **修法**：做 AI 产品时先问"这个产品的 AI 判断力差异化在哪里？"→ 把核心开发时间放在那个部分 → 视觉效果只要"够用"（达到基准线）就行，不要追高于 AI 判断力本身的投入比例。

## UU-3：功能难度 ≠ 用户感知价值，两者是独立维度

- **实锤**：效果图是技术难度最高（涉及图像编辑 API 或 AR 渲染）、翻车概率最高的功能，却在 5 小时黑客松里被优先追求，原因是"感觉最酷"。结果难度最高的功能翻了，拖累了整体产品印象。
- **机制**："酷"的直觉把难度高 × 视觉炫 当成了价值高的代理信号。但黑客松评委更看 AI 洞察力 + 产品逻辑，视觉特效的加分远低于直觉预期。功能难度是技术维度；用户感知价值是产品维度；两者正交。
- **修法**：列功能清单时，对每个功能标两个维度：技术难度（低/中/高）× 用户感知价值（低/中/高）。优先实现「低难度 × 高感知价值」的功能，最后才碰「高难度」功能——如果时间还够的话。

## UU-4：约束（"我有什么"）比需求（"我想要什么"）更先决定解空间

- **实锤**："只有 Gemini API key，没有 OpenAI API key"这个约束，在做了 Canvas + Gemini 图生图两个失败方案之后才被迫摆出来。如果第一句话就声明可用工具，整个技术方向会立刻不同（直接跳到 MediaPipe 或放弃效果图功能），省掉所有返工。
- **机制**：人类自然沟通方式是描述"我想要什么"，但技术实现里"我有什么工具/API/预算/时间"才决定解的边界。约束的缺失在对话开始时是隐形的，代价却在方案选择之后才显现。
- **修法**：任何新功能 / 新项目开始前，先盘约束清单：可用 API keys、预算上限、时间窗口、技术栈。约束先于需求，不是补充。

## 一个立即可做的检查

拿下一个要做的功能，回答这两个问题后再动手：
1. **这个功能，用户的期望基准是什么级别？我们的实现能达到那条线吗？**（基准 > 实现能力 → 不做）
2. **做这个功能，需要哪些 API / 工具？我现在有吗？**（没有 → 先列替代方案，再决定做不做）

## 诚实反束

这是单场黑客松会话的观察，不是判决。追效果图可能在某些评委面前是正确的策略——"wow factor"有时候确实影响比赛结果。分析者本身是没被验证的验证器，"不做效果图整体更好"是单场观察，而不是普遍结论。
