# uu-39 — 上下文债务不可见:拿着热文找新工具,最大的浪费躺在自己配置里

> 背景:一场从"看到 pxpipe 热文(把文本渲染成图片省 token)想要一套省 token 工具"出发的优化会话。结局反转:最大收益不是装任何新工具,而是发现自己有 44KB 的收尾流程规则文件——只在会话结束时用,却被 harness 每场全量加载,每一个会话都在为它付费。这个反差暴露了两个此前完全不在意识里的盲点。

## 盲点 1:优化冲动来自外部热点,而不是自家账本

**实锤**:提问的形态是"给我一套省 token 的工具/skill"——预设答案是**装新东西**。而实际最大的收益(44KB/会话)是**搬走旧东西**:它在自己系统里躺了近两个月,不需要任何新工具,只需要一次 `mv`。

**机制**:外部文章自带显著性(salience),自家的成本结构是隐形的。系统里有月度磁盘债务扫描、有资产对账,但没有任何东西量过"每场会话的常驻上下文有多大"。**不可见的浪费不会触发优化冲动**——它只能等一篇热文碰巧撞进来,而热文指向的方向(装 pxpipe)和真正的病灶(自家常驻配置)可以毫无关系。

**修法**:把"常驻上下文大小"变成可见指标。一行命令:读最新会话日志首轮的 input + cache_creation + cache_read 三项 token 之和,纳入月度债务扫描,与上月比 delta。涨了说不出原因 = 上下文债务在堆积。原则:**先量自己,再看热文**——优化方向应该由账本产生,热文只提供技术选项。

## 盲点 2:治理了"内容",从没治理过"加载语义"

**实锤**:这套配置系统的内容治理是苛刻级的——每条规则带 sunset 日落条款、瘦身动作先过红队、memory 写入有 merge 纪律。但那 17 个收尾步骤文件建立时,没有人问过一句"这会每场加载吗"。更扎心的是:目录里躺着一个前次会话留下的加载机制测试文件,测试搭好了、没有回收机制,一直挂着,直到这场会话才被顺手收掉结论。

**机制**:harness 的加载规则是隐性的——不报错、不提示、账单上不单列。于是"文件放在哪个目录 = 什么时候被加载 = 什么时候被计费"这个维度**从未进入治理框架**。这不是治理不严,是不知道存在这样一根需要治理的轴:你可以把每一条规则的内容都打磨到完美,同时对它们的加载成本一无所知。

**修法**:新建任何会被自动加载的配置文件前,加一个一行判断:"这内容在多少比例的会话里被用到?"每场都用 → 常驻区;只在特定条件用 → 条件加载或按需目录。不需要新仪式,只需要让"加载语义"成为和"内容正确性"并列的决策轴。

## 诚实反束

以上基于单场对话,是假设不是判决。做出这份分析的,恰好是发现并修掉这些问题的同一个执行者——验证器本身没有被独立验证。另外"最大收益"的框架也可能高估了 token 成本在真实约束中的权重:订阅制下烧的是配额不是现金,44KB 的心理重量和财务重量并不相等。
