# uu-18 — 同一套 Claude，两台机器不是一个大脑：跨机器的治理不会自动跟着走

> 背景：一场从「多 agent 审计自己的知识库 / 配置」中途突然转向「在另一台电脑上配置同一个 AI 工具」的会话里挖出来的盲点——从「话题切换」+ 我把另一台当成「同一个工具」的行为里被看出来的，我当时自己没意识到。

## 认知盲点 ——「跨机器的治理不会自动跟着走」

- **实锤**：前半场在主力机上花大力气硬化治理（精简配置、修几十条知识条目、定决策门、归档死项目）；后半场问「另一台怎么开同一个工具的某模式」，全程把另一台当「同一个工具，给我快捷键就行」。
- **机制**：刚硬化的一切（配置 / 知识库 / 自定义命令 / 安全护栏）**只活在主力机的本地目录**。另一台读它自己那台的目录，默认空的或一份旧副本。两台 = 两个互不同步的大脑。
- **修法**：另一台要么用 git 同步主力机的配置 repo，要么承认它是「裸脑」，别指望它有那套规矩。

## UU ——「在最没护栏的机器上，开最危险的模式」

- **实锤**：我问的是「另一台怎么开『跳过所有确认』的模式（俗称 yolo）」。
- **机制**：yolo 危险，所以主力机配置里专门有「敏感目录自动加护栏」挡它。但护栏**也只在主力机**。另一台没同步 → 开 yolo = 危险开满、安全网为零。我以为「开 yolo」是个独立动作，没意识到护栏和危险住在不同机器上。
- **修法**：在任何非主力机开「跳过确认」模式前，先确认那台有没有同步护栏。

## UU ——「源头改对了，泄漏的副本没跟着改」

- **实锤**：我在源头把一个关于自己的事实改对锁死了。但同一次知识库审计抓到，**某个分析笔记里还写着这个事实的旧（错误）版本**，已躺 39 天没人发现。
- **机制**：源头改对 ≠ 清副本——错误版本早漏进 N 个派生文件。几百个知识条目里还有多少过期事实在躺着，没被审计撞到就不浮现，知识库在**静默累积错误**。
- **修法**：改任何「源头事实」后 grep 全库找旧副本清理；或定期跑一次全库事实一致性审计。

## 最高杠杆 1 条 + 立即可做检查

> 在另一台机器跑一句「这台有没有我的主配置文件」的检查（Windows PowerShell：`dir $env:USERPROFILE\.claude\CLAUDE.md`）。
> - 报「找不到」= 裸脑，治理一条没有 → 开危险模式前务必先同步。
> - 存在但日期很旧 = 两脑已分叉，先同步。

## 诚实反束

基于**单场对话行为**的推断，不是判决。我没法远程看另一台的真实状态——「两机分叉」「副本泄漏」都是**待那条命令验证**的假设。而且分析者本身是个**没被验证的验证器**：同一场审计里，自动 finder 就有 20-40% 的误报率，对盲点的判断同样可能错。
