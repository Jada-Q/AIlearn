# uu-34 · 内部事实修正不会自动传播到公开副本（+ 卖伞人淋雨）

> 来源：一场"借鉴别人 SEO/GEO 优化文章"的会话。原本只想抄作业，结果审计自己的站时挖出两个我完全没意识到的洞。

## 洞 #1（最高杠杆）：内部真相源 ≠ 公开副本，中间没有管道

我给自己建了一套业内算得上极致的防编造体系：CLAUDE.md 里大段诚实性规则、个人事实档案（user_facts）、/factcheck 命令——AI 在对话里想编我的资历，层层拦截。

然后这次审计发现：我自己网站上的 llms.txt、JSON-LD、页脚、站点标题，挂着**错误的资格声明挂了近两个月**。

时间线很说明问题：
- 5 月初：网站上线，当时的资料写法后来被判定不准确
- 6 月下旬：内部事实档案修正、固化（"对外只写 FP2"）
- 7 月初：才发现线上公开文件从没被回头改过

**机制**：我的诚实性体系全部作用于"输出端"——每次新对话里不许编。但事实修正之后，没有任何一步是"grep 所有**已发布**的公开资产"。内部改对了，公开面继续错着。而 llms.txt / 结构化数据恰恰是 AI 引擎直接抓取引用的——等于全网的 AI 在学习一份错误的我。

**修法**：身份类事实变更时，附带过一遍"公开资产清单"——deployed 网站、GitHub profile、X bio、公开 repo 的 README、名片类项目。事实修正的完成定义从"内部档案改了"扩大到"全部公开副本改了"。

## 洞 #2：卖伞的人自己在淋雨

我有 GEO 业务线（帮客户做 AI 检索优化），有一张收费级的审计 checklist，给自己的内容站做过全套基础件。

审计发现：我**交付给客户的两个网站**，robots.txt / sitemap / 结构化数据全缺。其中一个 72 次 commit，另一个光首屏动画就迭代了几十轮——但没有一个 robots.txt。

**机制**：可见层（视觉/动画）有即时的眼睛反馈，投入自然越堆越高；不可见层没有任何反馈回路，于是投入为零。我没意识到的不是"GEO 重要"（我在卖它），而是"**我的交付物里不含我自己卖的东西**"这个矛盾本身——它需要别人（这次是 AI 审计）从外面才能看见。

**修法**：把"可发现性清单"写成 ship 前的硬规则（7 件：robots + AI 爬虫放行 / sitemap / 结构化数据 / canonical 指正式域名 / llms.txt 事实核对 / 部署目录卫生 / 上线后同层验证），客户站无条件触发。

## 附赠观察：优雅降级会把故障藏起来

同场还抓到：一个动画库的 CDN 链接失效了很久,但因为降级逻辑做得好，页面一切正常——只是动画全没了，安静得没人发现。另一个站的 git push→自动部署管道断了 48 天，push 全进了仓库，线上纹丝不动。

降级和容错做得越好，越需要主动巡检——**故障不再报错，只会沉默**。

## 诚实反束

以上是基于单场对话的假设，不是判决。洞 #2 有一个替代解释：客户站交付在先、我的 GEO 能力成熟在后——那就不是盲点，而是"知识增长后没有 backport 旧交付物的机制"。但后者同样是一个洞。另外，做这份分析的 AI 本身也是一个没被验证的验证器。
