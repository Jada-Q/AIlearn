# uu-06 — 你的验证闸门只守"新建"，"改进/打磨"从缺口溜走

> 这条 unknown-unknown 从一次"帮我找几个项目做改进"的人机协作会话里浮出来。最终什么都没留下，唯一的产出是一条教训——所以它值得写下来。

## 背景

一个独立开发者（用 AI 当执行者）有约 100 个项目，并且**早就给自己装了一道闸门**：启动新产品前必须先回答"有人在用吗 / 值得做吗"。这道闸门是他清醒的自我设防——他知道自己会过度建造。

然后这次会话发生了：目标是"找几个现有项目做市场调查后的改进"。AI 探索、建了 3 个改进、合并其中一个、还做了整套 UI 重做——**全程那道闸门一次都没响**。直到开发者亲眼看见成品，问出"这跟直接用通用 AI 有什么区别"，才发现整个方向没有护城河，全部放弃。

## UU-1（最高杠杆）：闸门按"动作名"设，反射就从没被命名的动作溜走

- **实锤**：自我设防的闸门触发词是"build / 新建项目"。但这次的请求形态是"改进"和"UI 打磨"——语义上够不到闸门。于是一个无人验证的产品被加固、被重新设计，没有任何环节问"它凭什么存在"。视觉投资发生在价值验证**之前**。
- **机制**：建造和打磨给的是快、确定、可控、零拒绝的能力反馈；验证给的是慢、不确定、可能让你失望的反馈。便宜的反馈循环永远自动胜出。所以建造反射不会消失，它只会**绕过任何按动作名设的闸门**——build 被拦就走 improve，improve 被拦就走 polish。
- **为什么是 unknown unknown**：主观上"我在做改进，是正经活"；行为上"在给没护城河的东西加固"。闸门列得越具体（只写 build/quick），越是给反射留逃生口——而你看不见这个口，因为你以为闸门已经覆盖了"过度建造"。
- **修法**：闸门从"按动作名拦"改成"按投入拦"。**任何**建造 / 改进 / 打磨 / 甚至投入自己时间之前，先答一句：**"谁是那个具体的、想要它的真人？ta 付过钱 / 用过吗？"** 答不出名字 → 不开工。"改进"和"新建"一视同仁。

## UU-2（人机协作专属）：把"能干的执行者在快速推进"误当成"方向正确"的信号

- **实锤**：AI 执行者探索、建造、合并、重做 UI，一路顺滑且每一步都做了核查，"在前进"的体感很强——这种体感压住了"该不该做"的提问，直到沉没成本已经产生。
- **机制**：过去"做起来很麻烦"本身是一道天然摩擦，会逼你动手前先想清楚。AI 把这道摩擦抹平了——于是战略问题不是没被问，而是被推迟到沉没成本**之后**才问。执行者越能干，越会带着你冲过本该停下的那道闸门。
- **修法**：和快执行者（AI 或人）协作时，在**放行执行之前**人为插一道"护城河闸门"；并明确要求执行者主动把这道闸门提上来——因为执行者自己也会失职。本场的 AI 就没在动手前提出"这东西凭什么存在"，它做了个尽职的执行者，没做成合格的参谋。

## 一个立即可做的检查

拿你**正在进行的下一件事**（无论建造还是改进），现在就回答："谁是那个想要它的真人？ta 付过 / 用过吗？" 答不出名字 → 今天先别碰它，去找那个人。

## 诚实反束

这是单场会话的观察，不是判决。做这个分析的 AI 本身是个没被验证的验证器——而且它这一场就完整地犯了 UU-2 描述的失职。请把以上当作待证伪的假设，不是结论。
