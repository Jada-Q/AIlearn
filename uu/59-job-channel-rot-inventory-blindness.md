# uu-59 · 求职渠道在腐烂，而你的巡检清单没覆盖它

> 出处：一场求职应募全流程会话（简历 → 作品集扩容 → 渠道 profile 重定位）结束时的盲点复盘。

## UU-1 公开求职渠道副本腐烂，巡检清单没覆盖

**实锤**：求职平台 profile 上挂着一条一个月前就在简历里纠正过的错误能力表述（语言水平写高了）、一个错了整整一年的作品日期、和一个默认占位头像——期间 165 人访问过这个页面，其中包括正在选考我的公司的招聘负责人。而这一切发生在应募最密集的一周。

**机制**：这是「内部修正 ≠ 公开副本修正」（uu-34）的复发变体。我有一张"事实变更后必须巡检的公开资产清单"（网站 / GitHub / X / 公开 repo），但**求职渠道从来不在清单上**——清单式防线的死角在于：清单本身也会过时。上次修好了错误，只是修在了清单覆盖的地方。

**修法**：把求职渠道 profile 列入公开资产传播清单；能力/资格类字段变更时，全渠道 grep 旧表述。更一般的：每次靠清单躲过一劫时，问一句"这张清单上一次增补是什么时候"。

## UU-2 产出速度超过自我库存的检索能力

**实锤**：同一场会话里我两次要求"新建"其实已经存在的东西——让 AI"做一个作品集页"（我有一个已上线、几天前刚更新过的作品集站）、让 AI"加一个健身 app 项目"（它已部署，连名字都是 AI 从索引里替我挖出来的）。80 多个 repo 之后，我脑内的目录已经跟不上我的产出。

**机制**：**交互记忆（transactive memory）**——人会把"记住什么"外包给伙伴或系统，但它成立的前提是"知道去哪查、且真的去查"。我的项目索引和 dashboard 都存在，但不在决策回路里：感知上"让 AI 重做一个"比"检索我有什么"更便宜，于是库存越大越不查。

**修法**：不逆着习惯改自己——把规则装在 AI 侧：听到"新建/做一个 X"，先报同类已有资产再动手。这场会话里两次撞车都是被这个动作拦下的。

## 最高杠杆

UU-1。检查动作（当天可完成）：打开求职渠道 profile，改掉错误表述、换掉默认头像、修正作品日期——十分钟，覆盖的是选考期间每一个来访者的第一印象。

---

*诚实反束：单场观察非判决；这场会话的大部分动作由 AI 执行，我的行为样本主要是指令流——分析者本身是一个未被验证的验证器。*
