# uu-43 — 校准发生在错误的执行环境 + 教训从不回扫存量

> 背景：一场"Telegram 天天推 6 项自动化告警"的诊断会话。查明 6 项里只有 1.5 项是真故障，其余是监控系统自身的误报——而误报的根因，本身就是两个此前完全没意识到的盲点。

## 认知盲点：告警疲劳稀释真故障

- **实锤**：健康体检推送至少连发 5 天才被拿来认真查。这期间，真死了 26 天的两个周任务和其实完全健康的一个日任务，混在同一条消息里同权重展示。
- **概念点名**：警报疲劳（alarm fatigue）——医疗/运维人因工程的公认概念：暴露在高频重复告警下，人的响应率随时间衰减，真信号跟着陪葬。
- **修法**：告警消息区分「新增 vs 持续第 N 天」；同一告警连续 N 天升级到不同通道，而不是原样重发。

## UU-1：校准发生在错误的执行环境

- **实锤**：监控系统 5 天前做过一次探针改造，改造后新探针在终端环境完全正常，在真实执行环境（launchd 定时任务，权限模型与终端不同）**从上线第一天起 100% 失败**，连续 5 天无人察觉——哪怕"新验证工具上线前必须先做阴性测试"早已是固化规则。
- **机制**：阴性测试在开发环境（终端）做完就算过了，但终端和 launchd 的权限模型是两个宇宙（macOS TCC 对二者的 Desktop 授权完全不同）。验证过的是"另一个宇宙里的脚本"。（查不到公认术语，此处只做机制描述。）
- **修法（检查动作）**：定时任务的上线校准必须由调度器本身触发一次（launchctl kickstart + 核对最终产物），终端跑通不算数。

## UU-2：教训只在新建时触发，从不回扫存量（最高杠杆）

- **实锤**：知识库里早就记着"bash 子进程会被 TCC 拦在 Desktop 外，新项目别放 Desktop"这条教训，但三个核心自动化的数据源全部留在 Desktop 子目录雷区里——教训被记住了，却从没拿去照一遍已有系统。
- **机制**：规则的触发器是"下次做新东西"，存量布局没有任何机制触发重审。教训被归档成"已知坑"，而不是"待清雷区"——归档动作本身给了大脑一个"已处理"的错觉。
- **修法**：每记下一条环境类教训，追问一句"现有系统里有几处正踩着它？"——一条 grep 就能列出全部踩雷点，10 分钟的一次性动作。

## 为什么 UU-2 杠杆最高

UU-1 当场被硬修复了（授权 + 监控加了权限判别分支），UU-2 的雷还埋着：授权本身会静默变化（这次的证据文件是 0 字节，分不清"授权曾有后丢"还是"从未有过"——两种解释都证明授权状态不可假设恒定），数据不搬出雷区，下次系统大版本更新同坑重演。

## 诚实反束

单场观察非判决；本场分析者（Claude）同时是修复方案的执行者，既当运动员又当裁判，"最高杠杆"的排序未经第三方验证。会话中一度断言"授权约一个月前丢失"，那是推测非实证，本文已改为存疑表述。
