# uu-17 — 探索快感盖过需求验证：为"自用工具"越投越深，需求却只验证了一下午

> 背景：一场从"用 video-as-code 工具做产品 promo"滑向"自建数字分身工作流 + 装离线开源数字人 + 给另一台电脑装开发环境"的长会话里挖出来的盲点。挖的是作者本人的行为，不是代码。

## 🔴 最高杠杆：技术栈越投越深，底层需求只验证了一下午

**实锤**：一次会话里，决策链从"能不能做数字分身" → "装离线数字人框架 + 差点卖掉一台电脑又决定留下 + 给它装开发环境"，越走越重。但"我需要数字分身讲产品"这个需求当天才冒出来，**一条真视频都没发出去验证过**。

**机制**：vibe coder 的"探索 / 搭建快感" >> "需求验证纪律"。"我能把它搭起来" 被悄悄误当成 "我需要它"。最讽刺的是——作者自己的规则库里写着一条 dogfood 硬门槛（自己用过 7 天原型才动手做），却在"自用工具"这个类别上把这道门自动绕过了：门对外（产品 / 客户）生效，对自己的玩具放行。

**修法 / 立即检查**：在投入本地基建之前，先用现成免费工具做 3 条真东西发出去，看反馈 + 看一周后自己还想不想做，再决定要不要建基建。**基建随时能补，方向错了全白搭。**

## 🟡 用"换下一个工具"应对结构性约束，搜多少都绕不开

**实锤**：付费云端 → 免费云端 → 开源 → 按量 API → 离线开源，问了一圈，每个都撞上同一面墙（要么花钱、要么要专用显卡）。这是一个物理"三选二"（免费 / 质量 / 省事），不是"再找一个就解决"。

**机制**：面对结构性约束，第一反应是"水平搜索更多选项"，而不是"识别这是不可绕过的约束、然后决定接受哪个代价"。

**修法**：当出现"找了好几个都不行"→ 停下来问一句"它们是不是卡在同一个约束上？"先把三角摊开再选，而不是继续找第 N 个。

## 🟢 差点卖掉的资产是唯一可行解：对"手头已有的东西"无感知，默认向外采购

**实锤**：一路在问免费在线、租云端算力，直到被反问"你有没有带专用显卡的机器？"，才想起一台**正在挂卖**的旧台式机——而它恰好是整个目标唯一现实可行的解。

**修法**：任何"我需要 X 能力"的问题，先问一句"我手头 / 已经买过的东西里，有没有现成的？"再向外找。

## 诚实反束

以上是单场对话的观察，不是判决；做这个分析的 AI 本身也是一个没被验证的验证器。而且要诚实：🔴 那条盲点里，AI 助手是**同谋不是旁观者**——整场一路在帮用户往"装工具"推进，每一步都在 enable，没有任何一步停下来问"需求验证了吗"，直到复盘才回看。这对 AI 助手同样是提醒：过度殷勤地帮用户往前推，会和用户一起跳进同一个兔子洞。
