# uu-44 — 你整理的是文件，喂养混乱的是制度:清理 state，不动 generator

> 背景:一场桌面大整理（76 个顶层项 → 41 个）挖出来的。整理本身很顺利——盲点藏在"整理请求的形状"里。

## 实锤

三个行为对照，指向同一个洞:

1. **两次整理请求（"整理桌面"、"视频太多整理一下"）都是"把乱的收好"**，从未包含"为什么会乱"。
2. **三套并行的归档制度**（一个 /kill 命令写入的坟场 + 两个手动时代的 Archive）共存数月、**同主题文件夹六处开花**——这些是制度性乱源:每次收尾临场发明一次仪式、开新文件夹的成本永远低于找旧文件夹。但发现并点破它们的是 AI，本人看到分析后才说"合并它们"。
3. **被取代的旧版本从不标记"已废"**——同一个 demo 视频的三档画质版本并排躺在桌面，堆到"太多"才触发清理。

## 机制

整理被当成**清洁行为**（把现状恢复整齐），而不是**系统设计行为**（改掉持续产生混乱的规则）。只清理 state、不改 generator 的后果是数学般确定的:三个月后同样的乱再来一遍，永远循环。

**概念点名:转移负担（Shifting the Burden）**——Senge《第五项修炼》里的系统基模:症状解（清理）见效快，于是一次次替代根本解（改制度）;而且每用一次症状解，做根本解的动力就再弱一分。大扫除的爽感，恰恰是制度改造最大的敌人。

## 修法

每次说"整理 X"时，强制补一问:**"是什么在持续产生这种乱？"**

答案只有两个合法去向:要么落成一条制度（本次案例:三个归档目录合并为一个、在里面放一份 README 写明"死项目只进这里、由哪条命令写入、怎么复活"），要么显式接受"这类乱就是会定期产生，定期清即可"——接受也是决策，默认循环不是。

**最高杠杆的微习惯**:开新顶层文件夹之前，先看一眼已有的，问"哪个能装它"。"开新文件夹的冲动"就是六处视频文件夹的 generator 本体。

## 正向对照

同一天里也有反例值得记:视频画质不满意时，这位创作者没有说"再录清楚一点"（同层重试），而是问"能不能**从代码层直接编译视频**"——自发的换层思维，质疑生成方式本身而不是在旧方式里加码。同一个人，同一天，在一个领域里完成了另一个领域还没完成的跃迁——盲点从来不是能力问题，是注意力的照射角度问题。

## 诚实反束

基于单场对话加一次桌面考古的观察，不是判决;"她只是没把制度改造当成可以委托出去的活"是同样成立的另一种解释;而且分析者本人恰好是这次清理的执行者，有动机放大"制度洞察"的价值。分析者本身也是没被验证的验证器——请打七折读。
