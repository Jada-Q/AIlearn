# uu-16 — 把"开发工具"当生产后端：CLI 当 runtime 的隐形雷

> 背景：一场调试对话——一个 vibe-coded 的 macOS 现场翻译 app 翻译全挂，逐层排查 4 层 bug，最终定位到它调用的命令行工具在运行时启动了一堆无关的后台服务、把自己卡死。盲点不在那 4 个 bug 里，在"为什么会有这 4 个 bug 排成一队等着"的结构里。

## 认知盲点：把交互式 CLI 当成产品的运行时后端

我的翻译 app 直接在每句话的热路径上 spawn 了一个命令行 AI 工具（`claude --print`）当翻译引擎。

**实锤**：app 的翻译函数每收到一句话，就起一个 `claude` CLI 子进程跑翻译。能跑通的 demo 给了我"它就是个 API"的错觉。

**机制**：这类 CLI 是为**交互 / agent** 设计的**重启动程序**——每次调用都会读配置、启动你配置过的所有插件服务（MCP server）、跑鉴权、可能加载钩子。把它当成"轻量 API"塞进 runtime，等于每句话都付一次"启动整个 agent"的成本 + 吃下所有插件副作用。

## Unknown unknown：副作用是跨时间、跨项目延迟引爆的

真正可怕的不是慢，是**因果链被时间和项目隔断**。

- **实锤**：某天我给这个 CLI 加了一个浏览器自动化插件（playwright MCP）。半年后，在一个**完全不相干的项目**（翻译 app）里，这个插件被 CLI 默默拉起、启动浏览器、卡死整条翻译。两件事中间隔了半年和一个项目，我做梦也不会把它们联想到一起。
- **机制**：CLI 的"全局配置"是一个**隐形的全局可变状态**。你在 A 场景的一次配置改动，会无声地改变 B 场景里每一个 spawn 它的程序的行为。配置变、CLI 升级、鉴权过期——任何一个都能让这个 runtime 依赖**无声打挂**，且报错信息永远指不到真因（我的 app 三次都报"翻译引擎都挂了"，三次真因全不同）。
- **同根**：这是我半年前另一次盲点的同一棵根——分不清"开发期工具"和"运行期依赖"。开发工具图方便、可人工兜底；运行期依赖要轻、要可控、要能无人值守地稳定。拿前者当后者，迟早炸。

**修法**：runtime 调用走官方 API 直连（轻、快、无插件副作用、可控）；非要复用 CLI，就必须显式锁死它的配置加载（"只用我指定的、忽略全局配置" + 超时 + 健康检查），把那个隐形全局状态钉死。

## 元层盲点：验证"修好了"如果和故障不在同一个环境，等于没验证

- **实锤**：排查中 AI（也包括我自己）两次宣布"已修复、已验证"，都是真心的，但都基于**终端环境**的测试——而终端恰好用一种我没意识到的方式（输入是键盘、插件已被其它窗口预热）掩盖了**图形 app 环境**里的真 bug。是我一句直觉的"你自己检查一下能正常运行吗"，才逼出了在真实环境的验证，挖出真凶。
- **机制**：验证者和被验证系统共享同一套乐观假设时，验证就是走过场。"我测过了"必须追问一句"你测的环境，和它出故障的环境，是同一个吗？"

## 最高杠杆 1 条 + 立即可做

不是去修更多 bug，是去**清点同类雷**：grep 自己所有项目里调用这个 CLI 的地方，看还有几处把开发工具当 runtime 依赖埋着同一颗雷。一处一处换成 API 直连或锁死配置。

## 诚实反束

单场观察，不是判决。写下这条分析的 AI 本身，就是个没被验证的验证器——它在本场前两次就误判过"修好了"。所以它对"我的盲点"的判断，同样该打一个折扣再听。
