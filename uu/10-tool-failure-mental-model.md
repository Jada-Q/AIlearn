# uu-10 — 工具失败 vs 系统推送：两种心理模型的诊断分岔

> 来源：一次调试 macOS 本地工具链的会话，YouTube 下载脚本失败后日志文件自动弹出，被误读为"系统在发报告"。

---

## 认知盲点：弹出的文件是谁发的？

macOS 工具失败后常见的 error handling 是 `open "$LOG_FILE"`——让系统用默认 app 打开日志文件，让用户看到错误。这个行为的外观是：一个文件窗口突然出现。

用户的解读是："有什么东西在**发**给我这个报告。"

实际机制是："我的工具失败了，它打开了自己的错误日志。"

这两种解读触发完全不同的诊断路径：

| 解读 | 第一步去查的地方 |
|---|---|
| 系统在推送 | LaunchAgent / cron / Telegram 推送 / 定时任务 |
| 工具失败了 | 错误日志 / exit code / 触发条件 |

这次的答案在第二列，但直觉带着往第一列走。

**实锤**：用户的原话是"为什么这几天会**自动发我**这个报告"——"发"字是心理模型的泄露，把 macOS 本地弹窗等同于外部推送。

---

## Unknown Unknown：自建工具链的"失败行为地图"是空的

积累了约 35 个 LaunchAgent + 各种 shell 脚本 + AppleScript droplet 之后，这些工具失败时的副作用是分散的、没有统一策略的：

- 有的：静默退出，什么都不显示
- 有的：`open "$LOG"` 弹出日志文件
- 有的：推 Telegram 消息
- 有的：重启服务（launchd KeepAlive）
- 有的：写 `/tmp/*.log` 但不通知

没有一张"失败行为地图"的结果是：同一类失败（脚本出错）会触发十几种不同的外显行为，无法预判"某工具出错了会怎么表现"。

**机制**：工具是分批建的，每个的 error handling 是当时就地决定的，从来没有作为一个整体来设计过。

**修法（最小侵入）**：不需要统一重写所有工具。在已有的 `automation-health.sh` 里加一项检测——"最近 N 天 `/tmp/*.log` 有没有异常新增"——把隐性失败变成可见的每日体检信号。

---

## 这次 bug 的具体根因（附加上下文）

AppleScript droplet 的 `looksLikeURL()` 函数只检查 URL 是否以 `https://` 开头，不验证 YouTube 视频 ID 是否存在。当剪贴板没有 URL 时，对话框默认填入 `https://www.youtube.com/watch?v=`（无视频 ID），用户直接按回车，`yt-dlp` 收到这个残缺 URL 失败，触发 `open "$LOG"`。

修法：在 `looksLikeURL()` 里加视频 ID 的存在性检查；默认填充值改为空字符串。

---

## 最高杠杆 1 条

下次工具出现"莫名其妙的弹窗或文件打开"，最快定位路径：

```bash
ls -lt /tmp/*.log | head -5
```

看最近改动的日志文件，几乎总能找到是哪个脚本在失败。不需要先排查所有自动化任务。

---

## 诚实反束

这是单场会话的观察，不是对某人认知结构的判决。可能在其他场合完全清楚 `open $LOG` 是什么，只是这次直觉带偏了方向——这是正常的，工具多了心理模型跟不上是默认状态，不是能力问题。分析者本身（LLM）也是没有被独立验证的验证器，这段分析可能本身就有盲点。
