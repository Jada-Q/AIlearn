# 20 — 省 token 从量自己的账本开始：热文教你装新工具，我搬走了 58KB 旧配置

> 起点是一篇热文：pxpipe，把 Claude 的文本上下文渲染成图片喂回去，号称账单降 59-70%。我顺着它研究了两天，最后一个新工具没装，省得却不比它少——因为真正的浪费不在"没用上新技术"，在我自己的配置目录里躺了近两个月，每一场会话都在为它付费。
>
> 这篇讲三层：计价原理（为什么图片能省钱）、套利层级（哪些省法能长期依赖）、实操五步（怎么在自己机器上复现我的审计）。所有具体数字都是我一台机器在 2026 年 7 月的实测，版本会变，以你自己量到的为准。

---

## 一、两套计价规则，一个脱钩点

**文本按"件"计费。** 模型把文字切成 token 收费，像快递按件计价。切法很吝啬：英文散文约 4 字符一件，但代码、JSON、工具输出这种"密文本"（全是符号、缩进、重复的 key），pxpipe 在真实 Claude Code 流量上的实测口径是约 1 字符 = 1 token（源：[teamchong/pxpipe](https://github.com/teamchong/pxpipe) README，2026-07）。

**图片按"面积"计费。** Anthropic 官方口径约为 宽×高÷750 个 token——一张 1928×1928 的图约 4700-4900 token，**不管这张图是一片纯白还是密密麻麻写满 9 万字符**。像快递按集装箱体积计费：箱子多大钱就多少，里面塞多满随你。

**脱钩点就在这。** 面积计价假设图片的信息密度是"正常照片"水平；你偏偏塞进一张 5 号字排满的小抄，密度是照片的几十倍，按面积只收一份钱。pxpipe 实测密文本转图后约 3.1 字符/token，对比文本通道约 1 字符/token——3 倍套利就是这么来的。模型读得出来，是因为它本来就有视觉能力（computer use 就是靠读屏幕截图工作的）。

**但它必然有损。** 文本通道是无损管道，每个字节原样进模型；视觉通道是**识别**不是解码——模型"看着像什么就是什么"，跟人认字一样。普通句子认错一两个字能靠语言规律猜回来；hash、密钥这种没规律的字符串猜不回来，而且**认错了它自己不知道，会自信地编一个**。pxpipe 自己的 README 写得很诚实：精确 12 字符 hex 串，Fable 5 读对 13/15，Opus 0/15，"misses are silent confabulations, not errors"。

抽象成一句话：

> **同一份信息有多种"表示"，每种表示的计价不同——永远选"最便宜的合格表示"。**

## 二、套利层级：哪些能长期依赖

按稳定性排序，从最稳的吃起：

| # | 通道 | 价格 | 性质 |
|---|---|---|---|
| 1 | 把重复的 LLM 判断固化成代码/hook | **0x** | 官方外——干脆不用模型 |
| 2 | prompt 缓存（前缀重复部分） | ~0.1x | 官方折扣 |
| 3 | 指针/按需加载（索引放钩子，全文用时才读） | 差距可达百倍 | 官方机制 |
| 4 | 表示格式（diff 不重发全文、表格代替逐行重复 key 的 JSON） | 数倍 | 无损技巧 |
| 5 | 分层模型（机械步骤用便宜模型/低 effort，判断步骤才用大模型） | 3-10 倍 | 官方折扣 |
| 6 | 输出侧（出口 token 是入口约 5 倍价——"只要结论/只回 diff"） | 5 倍杠杆 | 无损技巧 |
| 7 | Batch API（不急的任务半价；仅按量计费的 API 适用，订阅制无此通道） | 5 折 | 官方折扣 |
| 8 | 有损像素（pxpipe 类） | ~3x 但有损 | **定价漏洞** |

分辨器就一条：**官方给的折扣**（1-7）平台希望你用，放心建进工作流；**定价漏洞**（8）是面积计价没料到微缩小抄，属于错价，迟早被修——只做可随时拆掉的外挂（pxpipe 是本地代理，一行环境变量就能摘，这个架构选择本身是对的）。

还有一条反向套利容易被忽略：**套利方向不固定**。密文本走图片便宜，但 UI 状态走文本比走截图便宜——一张全屏截图上千 token，Playwright 的 accessibility snapshot（页面结构文本）往往几百 token 还无损。哪边便宜走哪边。

## 三、我的审计：钱到底漏在哪

带着上面的框架回头量自己，发现三处漏，全部无损可修：

1. **44KB 的收尾流程规则**，放在会被 harness 每场自动加载的目录里——但它们只在会话结束跑收尾时才用。每一场跟收尾无关的会话都在为它全额付费。修法：整体搬到按需目录，调用方本来就按路径显式读它们，功能零损失。
2. **记忆索引超了加载上限**。实测 MEMORY.md 有 200 行的加载硬上限（按行数，不按字节——我先按字节压了一轮才发现量纲搞错了，警告原文里其实写着 `limit: 200`）。超限的尾部条目每场"失忆"，等于既占地方又不干活。修法：分层索引——休眠项目条目拆到子文件，主索引留一行指针，一跳可达。
3. **只在特定流程用的长规则常驻**。修法同 1，外加一个实测彩蛋：规则文件头部写 `paths: ["**/*.css"]` 可以实现条件加载，只在会话碰到对应文件时才进上下文。

三处合计约 58KB 常驻内容移出了每场计费区。没有删除任何内容——全部只是搬家。

## 四、实操五步（可直接抄）

**① 量基线。** 每场会话的真实开销记在 Claude Code 的会话日志里（`~/.claude/projects/<项目目录>/` 下的 `.jsonl`）。取最新会话第一轮的 `input_tokens + cache_creation_input_tokens + cache_read_input_tokens` 三项之和——这是你开场就要付的"底价"，账本口径不是估算。我的基线是 98,486 token。

**② 盘点常驻区。** 这些每场自动全量进上下文：`~/.claude/CLAUDE.md`、项目 `CLAUDE.md`、`~/.claude/rules/` 下所有 md、MEMORY.md（前 200 行）、每个 skill 的 description。这些按需才读：`commands/`、`workflows/` 下的文件正文。跑 `wc -c ~/.claude/CLAUDE.md` 和 `ls ~/.claude/rules/` 看看你的常驻区有多重。

**③ 一句话搬家判断。** 对常驻区每个文件问："这内容在多少比例的会话里被用到？"每场都用 → 留下；只在特定流程用 → 搬到按需目录，原位置留一行指针（引用它的地方记得改路径）。

**④ 索引瘦身。** 如果你维护记忆索引且超过 200 行：不要删条目，做分层——低频条目拆到子索引文件，主索引留一行"提到不在本索引的名字 → 先读子索引"。验证方法：新旧文件的链接并集 diff 必须为空。

**⑤ 验证，别信自己。** 两层：
- 软验证——开个新会话贴这段（关键是**逼引用，不问有没有**，没加载的内容它引不出来，是/否它可能顺嘴答）：
  > 不读任何文件不用工具，只凭已加载内容：你的规则里有没有字符串「<搬走文件里的独特字符串>」？有则原样引用那一行；没有回答"没有"。
- 硬验证——重复 ①，新基线减旧基线就是省掉的数，API 实际计费口径。

**使用场景对照**：重度 Claude Code 用户（五步全适用）；配置攒了几个月的老用户（②③④ 收益最大）；自动化管线构建者（额外吃第 5 条分层模型）；订阅制用户（省的是配额额度，逻辑不变）；API 按量计费用户（额外解锁 Batch 半价）。

## 五、真正的教训：不可见的浪费不触发优化

我有月度磁盘债务扫描、有资产对账，但没有任何东西量过"每场会话的常驻上下文有多大"。**不可见的浪费不会触发优化冲动——它只能等一篇热文碰巧撞进来，而热文指向的方向（装新工具）和真正的病灶（自家旧配置）可以毫无关系。**

所以这套操作的最后一步是把它变成仪表：我给月度债务扫描加了第三类"上下文债务"——自动量最新会话首轮 token 三项和，和上月快照比 delta，涨了说不出原因就去查常驻区最近塞了什么。配套一条对自己的纪律：**先量自己，再看热文。优化方向由账本产生，热文只提供技术选项。**

## 边界与诚实

- 所有数字（44KB / 58KB / 98,486 / 200 行）是我一台机器 2026 年 7 月的实测；加载机制和上限没有官方文档承诺，随 Claude Code 版本可能变化，照抄前先在自己机器复现第 ① 步。
- pxpipe 的 59-70% 是其 README 自报口径，我没有独立复测；它的有损风险描述引自其原文。
- 订阅制（Max）下这些优化省的是配额不是现金账单，心理收益和财务收益不相等——但配额吃紧时逻辑完全一样。
- 本文方法全部无损（内容只搬家不删除），但搬家会打碎一次 prompt 缓存（下场会话首次请求全价重建），一次性成本。
