# 路由 > 切换：带状态的并发资源不要原地切换

> 2026-05-28 · 打字翻译方向切换失败 3 次的复盘

给现场翻译工具加了一个打字翻译窗口，要支持「中→英」和「中→日」两个方向切换。**切方向、译文不变**。改了 3 次都没修好。第 3 次失败后停下来重看，发现**不是 fix 错了，是范式错了**。这条教训值得单独记 —— 它和 [08 物理天花板](./08-reframe-at-physical-ceiling.md) 是不同维度，那篇是认知层（什么时候不该再优化），这篇是工程层（带状态的资源该怎么管）。

---

## 失败的 3 次（其实是同一种思路）

| 尝试 | 想法 | 结果 |
|---|---|---|
| ① `onChange(of: config)` | 改 mode → 改 config 值，让 translationTask 自动重建 session | 译文不变 |
| ② nil-then-set | 先把 config 设 `nil` 再设新值，强制让 translationTask 拆掉旧 session | 译文不变 |
| ③ `.id(mode)` 重建子视图 | SwiftUI 销毁+重建整个翻译子视图，连带 translationTask 也重建 | 译文不变 |

3 次都在做同一件事：**单 session，想办法把它切换到新方向**。

---

## 真根因（实证 vs 推测，必须分清）

**✓ 实证：`AsyncStream` 是 single-consumer**

我用一个 `AsyncStream<String>` 让翻译闭包 `for await` 消费输入文字。Swift 文档明确说这种 stream **只能被消费一次** —— 一旦一个 `for await` 拿了它的 iterator，再起新的 `for await` 行为不保证。`.id()` 重建子视图后，新闭包想再消费同一个 stream，**拿不到任何东西**。所以即便 session 真的重建了，新打的字也进不了新闭包。

**⚠ 推测：`TranslationSession` 可能"粘"在首次 config**

Apple Translation framework 闭源，我没法证实。但从行为看，即使 config 变了，session 似乎仍按首次创建时的方向翻 —— 这能解释 onChange 和 nil-then-set 也失败。**这一条是推测，不是结论。**

实证那一条（① single-consumer）已经足以让"单 session + 重建"这条路走不通。如果推测那一条也成立，是双重坑。

## 双 session 一次成

不再尝试"切换"，而是：

```
[一个 translationTask 固定中→英，永远 for await enRequests]
[一个 translationTask 固定中→日，永远 for await jaRequests]
```

两条 translationTask 永远活着，各 own 一条 stream 的唯一 consumer。
切方向 = 改"打字时把字塞进哪条 stream"。**不动 session、不动视图、不动 stream 的 consumer**。

绕开了 single-consumer 坑（每条 stream 只有一个、永久的 consumer），也绕开了 session 可能黏 config 的问题（每个 session 从生到死只服务一个方向）。

---

## 通则：路由 > 切换

**凡是带内部状态的并发资源 —— stream、session、连接、订阅、缓存、actor 实例 —— 需要在几种模式之间切换时**：

> **预先创建几份固定资源 + 在入口路由**
> **比"共享一份资源 + 切换它的状态"稳得多。**

为什么：带内部状态的资源**一旦在运行就有 in-flight 状态**（队列里没消费的消息、连接的 handshake、订阅的位置、缓存的脏数据），"切换"几乎总是踩各种 race / 单消费者 / 状态机非法转换的坑。"多份并存 + 路由"让每份资源**生命周期里只服务一种模式**，没有切换、没有 race。

---

## 怎么识别该换范式（不是该调参数）

**同一模块 ≥3 次"调参数式"修复都没好** → 多半是范式错，不是参数错。停下来问：

1. **这个功能为谁存在？不存在会怎样？**（确认问题是真的）
2. **同类问题别处怎么做的？**（项目里、行业里）
3. **如果别处用 A 模式 work、这里你为什么用 B？**

我这次的红脸时刻：**项目里的语音翻译 CaptionBar 早就是双 session 模式工作了**。做打字翻译时我**没复用那个验证过的模式**，反而原地造了"单 session 切换"的轮子。等于把验证过的答案放在隔壁视而不见。

这种「同一个 codebase 里、隔壁就有可用范式而不见」的情况比单纯的不知道更危险 —— 它来自惯性：「这是新功能，我新写一套」。

## 适用边界（什么时候 "切换" 可能更好）

路由不是万能的。预创建 N 份资源 → 内存 N 倍。如果：

- 资源种类多到不能预创建（成百上千的任意配置 / 用户输入决定）
- 资源**创建极昂贵**（如加载 GB 级模型）
- 切换非常罕见（每天一次）

—— 这时"切换 + 复用"可能更合理，但**必须接受切换路径上要写更复杂的状态管理**（清队列、等 in-flight 完成、原子替换、单元测试覆盖切换）。

我这个案子是**有限种类（2-5 种语言对）+ 创建便宜（系统 session）+ 切换频繁**，三项都强烈指向"路由"。我却本能选了"切换"。

## 一句话

下次遇到「我有一个共享资源，想让它在几种模式间切换」的需求 —— 停一下，问："是不是应该多份并存按需路由？"
