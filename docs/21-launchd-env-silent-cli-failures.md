# 定时任务里的 AI 助手为什么半夜罢工:换个执行环境,你的脚本就是另一个脚本

> 背景:把一个手动跑了两个月、从没出过错的 AI 情报采集脚本改成 macOS launchd 每周自动跑。首次自动运行当场失败,而且失败得非常安静。修复链一共三个坑,每一个都属于同一族:**手动跑一万次也测不出来的错误**。

## 坑 1:定时任务是个"空房间"——缺一个环境变量,AI CLI 就不认识你

**现象**:launchd 触发的脚本里,`claude` CLI 报 `Not logged in · Please run /login`。但同一台机器、同一个账号,终端里跑一切正常。

**根因**:launchd 环境不继承你终端里的环境变量。这个脚本的 plist 里给了 `HOME` 和 `PATH`,但**缺 `USER`**——而 claude CLI 取登录凭证需要它。终端里 USER 天然存在,所以这个依赖在手动模式下永远不可见。

**定位方法(比答案更值钱)**:变量控制实验。用 `env -i` 起一个纯净环境,先照抄一个"已知能工作的定时任务"的全部环境变量复现失败,再一个一个加变量——加到 `USER` 时立刻成功。一次对照实验,替代对 PATH/权限/钥匙串的所有猜测。

**修法**:定时任务 plist 的 `EnvironmentVariables` 显式给 `USER` + `LOGNAME`。

## 坑 2:报错走 stdout,你盯着的 stderr 是空的

**现象**:脚本把 claude 的 stderr 单独存了日志文件,失败后去查——文件是空的,0 字节。看起来像"什么都没发生"。

**根因**:claude CLI 的 `Not logged in` 错误**打印在 stdout**,不在 stderr。按 Unix 直觉"错误去 stderr"布置的观测点,正好整个错过。

**修法**:诊断外部 CLI 失败时,stdout 和 stderr 两路都要留、都要看;失败日志里同时打印两者的摘要。

## 坑 3:`set -e` 悄悄杀死了你精心写的重试循环

**现象**:脚本里有一个写得很规矩的 3 次重试循环(失败记日志、sleep、重试)。但首次失败时,连"attempt 1/3 failed"这行日志都没出现——脚本直接死了。

**根因**:脚本开头有 `set -euo pipefail`(报错即停)。在它治下,裸写的命令一失败,整个脚本当场退出——后面的 `rc=$?` 和重试循环是**永远执行不到的死代码**。而手动跑时命令从不失败,所以这个 bug 潜伏了两个月零暴露。

**修法**:`set -e` 的脚本里,凡是想捕获退出码做分支/重试的命令,必须挂逃生口:`cmd || rc=$?`。写 `rc=$?` 之前,先抬头看一眼文件顶上有没有 `set -e`。

## 这三个坑的共同结构

它们都不是"代码写错了",而是**换了执行环境,同一份脚本就是另一份脚本**:

- 环境变量集不同(坑 1)
- 观测习惯失配(坑 2)
- 潜伏路径首次被触发(坑 3)

所以"手动测试通过"对定时任务几乎不构成证据。真正的验收只有一种:**用目标环境本身跑一次**(launchd 用 `launchctl kickstart` 手动触发排班任务),看到产物落盘、内容正确,才算装好了。

## 带走的检查清单

1. plist `EnvironmentVariables`:`USER` / `LOGNAME` / `HOME` / `PATH` 四件套
2. 失败诊断:stdout + stderr 两路都看
3. `set -e` + 重试:逃生口 `|| rc=$?`
4. 上线验收 = `launchctl kickstart` 真实跑一次到产物,不是"加载成功"

## 诚实边界

以上基于 2026-07 一次真实修复(macOS launchd + Claude Code CLI),坑 1/2 的具体行为属于该 CLI 当时版本的实测表现,版本升级后可能变化;坑 3 是 bash 语义,不随版本变。
