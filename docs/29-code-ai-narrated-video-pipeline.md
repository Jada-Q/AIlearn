# 用代码 + AI 生成解说视频的完整管线

> 2026-07-18 · 从"录屏为什么糊"这个反直觉洞察，到用 Gemini 出图 + Veo 动画 + ffmpeg 拼一支带中文解说的科普片，全程代码驱动

一天里我给一个学习工具录了 demo、把它做成带解说和 BGM 的成片、又用 AI 生成画面做了一支民俗风科普片。回头看，这些活儿的共性是：**画面可以千变万化，但"清晰、对轴、讲完"这些确定性的东西，必须用代码死磕，不能交给会飘的工具。** 这篇把踩过的坑和管线记下来。

---

## 一、先记住这条：清晰度是采集问题，不是编码问题

最反直觉、也最多人搞错的一点：

> **浏览器录屏（screencast）的分辨率 = CSS 视口宽度。把 deviceScaleFactor（DSF / 放大镜）调多高都不涨。只有 `page.screenshot` 才遵守 DSF。**

所以"视频糊"往往不是码率不够、也不是没上 4K，而是**源头就只录了 480 像素宽**，retina 屏上看当然软。事后 `scale` 放大只会更糊——放大不增加细节。

我用一个民间故事把它讲明白了，顺便也是那支科普片的脚本：

- **匠人把画画在一块小布上** = 屏幕（CSS 视口）
- **誊抄小鬼一根线一根线照抄** = 录屏（只誊抄布上有的织线数）
- **递上放大镜还是糊** = DSF 调高，screencast 不涨
- **织一块更大的布重画** = 用更宽的视口录制
- **一帧帧亲手重描** = 逐帧截图（帧步进）
- **把小画硬撑大只会更糊** = upscale ❌

### 出高清视频的三条路

1. **画面时间可驱动**（进度条 / 时钟能设值）→ 逐帧 evaluate 设状态 + `screenshot({scale:'device'})` 超采样。最锐。
2. **真实交互 + 布局能吃宽视口**（桌面 / 横屏）→ 直接把 `viewport` 设大（如 1920×1080）+ **DSF 保持 1** + recordVideo.size 与 viewport 相同（DSF≠1 时 recordVideo 会出灰边，翻过车）。
3. **真实交互 + 要保持窄 / 手机竖屏布局**（手机 app、朋友圈竖版）→ screencast 做不到（窄 CSS 宽 = 低清；宽 CSS 宽会把手机布局撑成平板双列）。**必须 DSF 帧步进**：窄视口 + 高 DSF，截图遵守 DSF → 手机布局 ×DSF 的无损帧。480 手机布局 × 2.25 = 1080 宽清晰帧，要 1440 就 DSF 3。

帧步进的实操：起一个并发截图循环，一边交互（真点击 + **瞬移滚动**，不用 smooth——低帧率下平滑滚动会顿），一边连拍，记录实测 fps 供组装用。~23fps、一段十几秒 ≈ 三四百帧。

### 编码旋钮（无论哪条路）

源头分辨率够高（宽视口或高 DSF）> CRF 16–18（越小越清，18 视觉无损）> preset slow（同 CRF 更清更小）> **scale 只降不升**。yuv420p 要求宽高都是偶数。

## 二、用 AI 生成画面：先定妆，再动画

想要非写实、有特定画风（我要的是平涂民俗重彩）的动画，**别直接文生视频**——文生视频容易跑偏成写实。正确顺序：

1. **图像模型出关键帧，定死画风**。把第一张当"画风锚点"传给后续每一张做参考，角色和配色才连贯。
2. **图生视频（image-to-video）animate**。视频模型会尊重输入图的画风，不会自己飘走。

### AI 视频/图像 API 的真实踩坑（2026-07 实测）

- **Imagen 4.0 对新用户下架了**（`imagen-4.0-generate-001` 等返回"no longer available to new users"），改用 Gemini 原生出图 `gemini-2.5-flash-image`（走 `generateContent`，返回内联 base64 图）。
- **模型 ID 必查不能凭记忆**。先 ListModels 看哪些真能调，别拿旧版本号硬试。
- **Veo 3.1 不支持 `generateAudio` 参数**（我本来就自己配旁白，去掉即可）。Veo 走 `predictLongRunning`：发起拿 operation name → 轮询 `done` → 从 `generateVideoResponse.generatedSamples[].video.uri` 下载（URI 要带 key）。
- **Veo 单段只有 8s**，而旁白往往二十几秒。用 **pingpong（正放+倒放）无缝循环 + loop 补时长**填满，比慢放或硬冻结自然。

### 想用付费 API 得先过一道网页门

一个从没用过 Model Garden 的新 GCP 项目，**光启用 API + 开计费还调不动** Imagen/Veo——publisher 模型对它不可见（403 not visible / 404 no access）。能真正调 Veo 的 key 要在 **AI Studio（aistudio.google.com/apikey）** 生成（绑 Google 账号那种），gcloud 建的项目 key 连基础 gemini 文本都 400 invalid。这一步是网页动作，脚本替代不了。

## 三、拼接：确定性管线死磕"对轴"

画面用 AI（敏态，要惊艳），**旁白 / 时长 / 字幕对轴 / 拼接用代码（稳态，必须精确）**。

- **旁白先行测时长**：`say -v "Panpan (Premium)"`（中文）逐段生成，量出每段秒数，**再按旁白时长去裁 / 补画面**——这样每句解说都压在对应画面上，而不是画面拍完再硬塞字幕。
- **字幕**：如果 ffmpeg 构建没有 `drawtext` / `subtitles` filter（我这台就没有），改用 **Chrome 渲染透明 PNG 字幕条**（`omitBackground:true`）再 overlay，中文排版还更可控。
- **封面**：`-disposition:v:1 attached_pic` 只是元数据缩略图，**播放时看不到**。要"打开就见封面"得把封面做成 **2s 开场帧 + xfade 淡入正片**；且 mux 单帧 png 封面时**别加 `-shortest`**（会把整片截到 0.03s，或把尾卡切没）。

## 四、成本（实测，非估算）

那支民俗风科普片，压成 3 个核心镜头（开场用静帧 Ken Burns 没动视频模型）：

- 图：4 张 × `gemini-2.5-flash-image` ≈ $0.12
- 视频：3 段 × 8s × Veo 3.1 Fast ≈ $3.6
- **合计约 $3.7**

对照当时查到的 Veo 定价：Veo 3.1 在 Vertex / Gemini API 上 Fast 档约 $0.15/s、Standard 约 $0.40/s、加原生音频再 +50%。压镜头数 + 开场用静帧，是把成本从 ~$8 砍到 ~$4 的关键。
