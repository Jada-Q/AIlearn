# 「修好了」的幻觉：一个空环境变量，骗过了两次修复

> 2026-07-17 · 一条 API 连续 401 了六场，被宣告"修好"两次，根因活到第三次排查才死——复盘这个只有一行字的 bug。

## 事故

我的个人复习库 API 突然开始拒绝所有自动化写入：401 Unauthorized。本地密钥文件在、内容没变，服务端却一律拒收。

这个故障存活了 6 天，横跨 6 场工作会话，其间被"修复"了两次——每次修复后都有笔记成功入库，看起来确实好了。然后下一场，又是 401。

## 根因（一行字）

服务端的环境变量 `LEARNLOG_API_SECRET` **存在，但值是空字符串**。

配置它的那天，变量名建了、值没写进去（echo 管道类手滑的经典形态）。而鉴权中间件写的是 `apiSecret && token === apiSecret`——空值判 falsy，这个 Bearer 通道**从出生那天起就没通过过一次**。

定位它只用了一个动作：把线上值拉下来做哈希指纹，跟本地比对。线上指纹是 `e3b0c442…` 开头——**空字符串的 SHA-256**，全世界最有名的哈希值。案子当场就破了。

## 真正值得写下来的：为什么被"修好"了两次

前两次修复做的事：清积压、重试、当次跑通了 → 宣告修好。

它们共同的缺陷是：**修复验证 = 症状消失，而不是根因被可复现地定位并排除**。当次跑通可能走了另一条鉴权通道、可能是环境差异——没人回答过"它为什么坏"，就宣告了"它好了"。

给自己（和 AI 协作者）的硬规则：

> **修复宣告必须含根因一句话**："因为 X 所以坏，我改了 X。"
> 说不出 X = 你还没修，只是又试了一次。

## 三个可带走的检查动作

1. **环境变量存在 ≠ 有值**——排查配置类故障时，`env ls` 看到名字不算数，把值拉下来做哈希指纹和源头比对（指纹比对不用把密钥暴露在对话/日志里）
2. **空字符串的 SHA-256 认脸**：`e3b0c44298fc1c14…`——看到它 = 你拿到的是空值，这条指纹值得背下来
3. **修复必须双向验证**：错 token 应 401、真 token 应通过——只测通过路径的"修复"，和没测一样

## 诚实反束

单案例复盘。"两次误诊"也有温和的解释：当时的目标是清积压而非排障，顺手宣告了修复——但这恰恰是问题本身：**顺手的修复宣告没有门槛**，而它的成本由未来的自己支付（这次是六场会话的断流）。
