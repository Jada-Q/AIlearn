# 从官方 algorithmic-art skill 到一个生成艺术系列：方法与四个真实 bug

> 2026-06-02 · 一次「读官方 skill → 照它的范式手写 → 演化成系列 → 推上 GitHub 带动态演示」的完整记录

起点是一个很小的问题：去 Claude 官方 `anthropics/skills` 仓库看有哪些 skill。其中一个叫 **algorithmic-art**，描述只有一句：

> "Creating algorithmic art using p5.js with seeded randomness and interactive parameter exploration."

顺着它做下去，最后长成一个叫 EMERGENCE 的三件生成艺术系列（差分生长 / 生成构造 / 奇异吸引子），上了 GitHub Pages + README 里带循环动图演示。这篇把方法和踩的坑记下来。

---

## 一、algorithmic-art 的三要素，每个都不是装饰

这个 skill 的范式可以拆成三个支点，缺一个体验就塌：

1. **p5.js** —— Processing 的 JS 版。它替你管好了 canvas 封装、`setup()`/`draw()` 动画生命周期、噪声/随机/颜色工具。画生成艺术不必从裸 canvas API 搓起。
2. **seeded randomness（种子随机）** —— `randomSeed()` / `noiseSeed()` 锁住随机序列：**同一个种子永远产出同一张图**。这是"作品"而非"一次性涂鸦"的关键——可复现才能收藏、分享、迭代。
3. **interactive parameter exploration（参数探索）** —— 把算法里的关键量（密度、噪声尺度、节点上限）抽成可调参数。创作过程从"改代码"变成"拧旋钮看反馈"，这才是生成艺术真正好玩的地方。

**结论**：照搬一个 skill 时，先搞清它每个要素解决什么问题，而不是只抄表面形态。

---

## 二、单文件零构建：刻意做薄

整个系列每件就是一个 `.html`：p5.js 走 CDN，没有框架（无 React/Vue）、没有打包工具（无 Webpack/Vite）。

为什么薄：

- 三个小作品上重型栈（React/Three.js）是杀鸡用牛刀，配置成本 > 作品本身。
- **无依赖 = 抗腐烂**。拖进浏览器就跑，十年后也不会因为某个构建工具的破坏性升级而打不开。生成艺术作为长期资产，这点比"工程规范"更重要。

判断口诀：**产物规模 < 工具链规模时，砍工具链。**

---

## 三、要"有新意"，换的是算法选型，不是参数

中途有个关键反馈：第一版（flow field 流场）和后来试的极光，被一句"不好看也没新意"打回。

根因不在调色，在**算法选型太大众**——flow field 和极光是生成艺术最被用烂的两个 demo。

**教训**：当作品"没新意"时，第一反应不该是调颜色/密度（那只是换皮），而是**换底层算法**。于是从 flow field 跳到差分生长（differential growth），形态完全不同，立刻有了辨识度。后续系列化也遵循同一原则：三件分别是**网（线）/ 塔（体）/ 缦（点）**，差异来自三种不同算法，而非三套配色。

## 四、四个真实 bug（症状 → 根因 → 教训）

### Bug 1：动画完全不刷新——p5 的 `redraw()` 只认 `draw()`

- **症状**：滑块改了、数值变了、标签也更新了，但画面纹丝不动。
- **根因**：用了 p5 的 `redraw()` 触发重绘，但渲染逻辑写在自定义的 `render()` 里。`redraw()` 只会调用 **p5 约定的 `draw()` 函数**，不认你另起的函数名。
- **教训**：用框架/库时，**必须走它约定的生命周期钩子**（p5 是 `setup`/`draw`/`redraw`）。自己另开一个同义函数，库不认，且**不报错**——最难查的就是这种静默失效。

### Bug 2：动态吸引子隔几秒变全黑——参数漂移漂进了退化区

- **症状**：让 de Jong 吸引子的参数随时间缓慢漂移产生流动感，结果每隔几秒整屏几乎变黑。
- **根因**：de Jong 吸引子有些参数组合会"坍缩成一条细线"（spread 极小）。漂移幅度设太大（±0.45），即使起始参数是好的，漂着漂着就撞进退化区。
- **教训**：**对参数空间不均匀的系统做动画，振幅要贴着"已验证的好点"走小步**。我把幅度收到 ±0.18，并在选起始参数时加了一道"spread 检测"自动排除退化解。连续系统 ≠ 处处可用。

### Bug 3：反应扩散被弃——质感不统一 + 卡顿，是"工具失配"不是"做得不好"

- **症状**：第四件试了反应扩散（Gray-Scott），每次换种子要算几秒、且柔边模糊，和系列里其他三件的锐利线条质感不搭。
- **决策**：直接砍掉，不硬救。
- **教训**：**一个元素再"对"，如果它的固有质感/性能和整体不兼容，就是错的选型**。砍掉比强行统一更省。识别"工具失配"（这算法本身不适合这个场景）和"做得不够好"（同一算法没调好）是两件事。

### Bug 4：构图偏上——不要把布局写成魔法比例

- **症状**：等距结构塔群整体飘在画面上方，下半屏大片空。
- **第一次修**：把 `originY` 从 `height * 0.40` 改成 `0.34`——治标，换个种子又偏。
- **正确修**：**实测整个结构的垂直包围盒，把中心对齐到画面中线**。
- **教训**：居中/对齐这类布局，**算出来**（测 bbox）比**猜比例**（写死 0.34）稳。魔法数字只对当前这一帧成立。

## 五、在 GitHub 上展示"动态"演示的工程做法

需求：让 GitHub 仓库首页就能看到作品在动。

**关键坑**：GitHub 的 README（markdown）**不会自动播放 mp4**——`<video>` 标签在 README 会被过滤掉。能内嵌并自动循环的只有**动图 GIF**。

做法：

1. **headless Chrome**（puppeteer-core 驱动系统 Chrome）逐帧录屏，每件录 40+ 帧。
2. **ffmpeg** 用 `palettegen` 两遍法合成 loop GIF（先生成调色板再应用，**防止暗底渐变出现色带**），缩到 640px 控制体积。
3. GIF 存进仓库 `assets/`，README 用 `![](assets/x.gif)` 嵌入 —— GitHub 自动循环播放。

**教训**：「在 README 里展示动态效果」这个需求，正确解不是 mp4 而是 GIF，且质量靠 ffmpeg 的 palettegen——不知道这条会做出一张色带斑驳的劣质图。

---

## 一句话总结

照一个官方 skill 做东西，价值不在"抄出一张图"，而在**吃透它的范式（p5 + 种子 + 参数）后能自己演化**：用算法选型换新意、用 bbox 而非魔法数字做布局、对退化敏感的系统小步漂移、砍掉质感不兼容的元素、并知道 GitHub 上动态演示只能走 GIF + ffmpeg palettegen。
