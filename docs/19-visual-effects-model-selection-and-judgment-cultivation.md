# 视觉效果选型、质量基准线与判断力培养：一次黑客松连续三次翻车的三层复盘

> 2026-06-13 · 5 小时黑客松，构建 AI 妆容顾问 app

做了一个黑客松项目：用户上传自拍 → 选场景（上班 / 约会 / 派对）→ Gemini 分析面部问题 → 呈现 BEFORE 标注 + AFTER 效果图。

AI 诊断部分很快跑通了，Gemini 2.5 Flash 读懂了脸，给出了有说服力的问题分析和色调建议。然后花了大量时间追 **AFTER 效果图**——连续三次技术方案翻车，最后的结论是：不做效果图，整体产品印象反而更好。

这个翻车过程值得记下来。它不只是"选错了 API"，而是同时暴露了三层判断的空白：**技术选型、产品设计、以及如何让这类判断可以迁移**。

---

## 一、三次翻车的完整过程

**第一次：Canvas 2D + MediaPipe**

用 MediaPipe Face Mesh 获取 468 个面部关键点，在 Canvas 上用 `mix-blend-mode: multiply` 叠加妆容色块。技术上跑通了，视觉上像"画笔涂上去"——完全不自然。

**第二次：Gemini 图像生成 API**

换 `gemini-2.0-flash-preview-image-generation`，把原图 + 诊断结果发过去，让 AI 生成效果图。API 调用成功，但返回的是一张全新生成的人脸——本人消失了。

**第三次：MediaPipe + 更精细的 Canvas 渲染**

回到 Canvas，加了高斯模糊边缘、按场景区分色调和透明度。效果比第一次好，但依然在"看得出是 AI 色块"的层级，没达到用户期望基准。

最终决定：不做效果图，只呈现 AI 诊断文字 + 色卡建议 + 步骤。整个产品的印象反而更清晰。

---

## 二、技术层：视觉任务按"是否保留原图身份"分两类

三次翻车的根因，在第二次最清晰：**把图像编辑问题误当成图像生成问题**。

这是两类截然不同的任务：

| 任务类型 | 要求 | 正确工具 | 错误工具 |
|----------|------|----------|----------|
| 文生图 | 生成全新图片 | Imagen 3、gpt-image-1 generations、gemini 图生图 | — |
| 图像编辑 | 改原图，保留人物身份 | gpt-image-1 edits（inpainting） | gemini 图生图 ❌ |
| 实时 AR | 浏览器内，<100ms | MediaPipe + Canvas/WebGL | 任何 server API ❌ |
| 专业级 AR 妆容 | 皮肤光学级别 | ModiFace / YouCam SDK | Canvas 2D ❌ |

**为什么 Gemini 图生图做不到"保留脸"**：扩散模型的工作方式是"加噪 → 去噪还原"。普通文生图只懂从噪声中生成一张新图；inpainting 模型在去噪时还需要接收一张 mask（哪里保留 / 哪里重画），用原图像素作为条件约束。Gemini 的图生图没有这个 mask 输入，所以本人的脸会被重画。

**为什么 Canvas 2D 做不到自然的妆容**：Canvas 2D 的颜色混合是数学叠加（multiply / screen），而真实皮肤是次表面散射（光进皮肤、折射后出来）。两者物理模型不同，无论怎么调参数，观感都是"贴上去的"而不是"长在脸上的"。

**MediaPipe 关键点 ≠ 妆容边界**：Face Mesh 的 468 个点是解剖学坐标（眼眶骨、嘴角肌肉止点），但腮红、眼影的视觉边界是化妆师的经验边界，两者不重合。这是效果"长歪"的根本原因，不是实现质量的问题。

> **选型判断流程：**  
> 这个视觉任务需要保留原图中的人物身份吗？  
> → 是 → 只有支持 inpainting 的模型（gpt-image-1 edits）  
> → 否 → 任何图像生成 API 可用  
> → 需要实时（<100ms）→ 只能本地运行，server 端 API 全不够快  
> → 想做专业级实时妆容 → 需要 ModiFace / YouCam 企业 SDK

## 三、产品层：质量基准线的两区间模型

这次最反直觉的结论：**不做效果图，比做一个烂的效果图，整体产品印象更好。**

这背后有一个模型，在做任何视觉功能之前都值得先想清楚：

**功能质量存在两个区间，不是线性的。**

```
用户期望基准线
──────────────────── ← 高于这里：加分
──────────────────── ← 低于这里：负资产（不是中性的"没有"）
```

低于基准的功能不是"有总比没有好"——它会让用户拿你的东西跟他们见过的最好版本对比，这种对比会传染到产品里其他本来不错的部分。

不同功能类型的基准线高低不同：
- **AR 视觉效果**：基准极高（用户的潜在参照是 Snapchat / 美颜相机）
- **AI 文字诊断**：基准相对低（没有太强的心理对比物）
- **操作流程**：基准中等

这意味着，进任何视觉功能之前，要先估两件事：**用户期望基准在哪里** × **我的工具能达到哪里**。达不到基准 → 不做，把时间给基准更低的功能。

> 扩展到产品设计层面：MVP 不是"做一半功能"，而是"每个做了的功能都在基准线以上"。

## 四、AI 产品的差异化在判断力，不在视觉包装

追效果图的时间远超打磨 AI 诊断质量的时间——但打磨完的诊断才是没人能复制的东西。

这揭示了 AI 产品的一个结构性特征：

**传统产品**：视觉 = 差异化（做得比别人好看就赢）  
**AI 产品**：判断力 = 差异化（看得比别人准才赢，视觉够用就行）

一个 AI 妆容顾问，如果诊断和通用搜索没区别，AR 效果做得再精美也没有留存理由。反过来，一句精准的诊断（"颧骨偏干，T 区偏油，适合哑光底妆 + 腮红局部提亮而非全脸打底"）有它自己的说服力，不需要视觉特效作包装。

AI 产品的资源分配逻辑因此应该反直觉：
- 核心时间 → AI 洞察的质量（prompt 设计、输出结构、诊断准确率）
- 视觉时间 → 只要达到基准线就停，不追求视觉领先

## 五、元认知层：判断力怎么培养

这次翻车的更深原因：在开始做之前，没有先建立"这类功能的质量基准在哪里"的感知。

判断力不是学来的，是**校准**出来的。有三层：

**第一层：接触最好的（建参照系）**

进任何新功能类型之前，先花 10 分钟作为用户体验这个领域最好的产品——不是技术研究，是感受。

这次如果在动手前先打开 Snapchat 用一次 AR 妆容滤镜，5 秒钟就能知道 Canvas 2D 和"用户期望的基准"之间有多远。

具体资源：AR 妆容 → Snapchat + BeautyPlus；图像生成 → Adobe Firefly + Ideogram；网站设计 → Awwwards + Godly；排版 → Fonts In Use + Typewolf。

**第二层：小实验 + 立刻对比（把感受变成校准）**

自己做一个版本，放在最好的例子旁边并排看。差距会立刻变得具体——不再是"感觉不太对"，而是"边缘的羽化差太多"或"皮肤质感是贴上去的"。

这个动作反复做，会在内部建一张地图：**哪类工具有哪条天花板**。久了不需要对比，一眼就能估出来大概在什么区间。

**第三层：预测 + 复盘（让经验真正复利）**

做之前写一句话预测："我预测这个方案会/不会成功，因为 X。"

做完之后对照预测，问的不是"什么出错了"，而是"这类方案在什么结构性条件下注定失败"——因为后者才能迁移到下次。

这次 Gemini 图生图失败，表层原因是"API 不支持图像编辑"，结构性原因是"文生图模型在训练目标上就没有 inpainting 能力，不是参数调整能解决的"。后者让我以后看到任何文生图 API 都能立刻判断"不能用于图像编辑"，不需要再试一次。

---

## 提炼：每次进新视觉功能前的快速校准

1. **先体验基准**：打开这个领域最好的产品，用 5-10 分钟，建立"用户已经见过的最好版本"的感受

2. **约束先于需求**：动手前先列可用工具清单（API key、时间、预算），再选方案——不是反过来

3. **基准线自检**：我的工具能达到用户期望基准吗？达不到 → 不做，把资源给基准更低的功能

4. **写一句预测**：开始前说"我认为这会/不会成功，因为 X"——做完后对照，提炼结构性原因

这四步加起来不超过 15 分钟，可以避开大部分"做了很久才发现方向错了"的情况。
