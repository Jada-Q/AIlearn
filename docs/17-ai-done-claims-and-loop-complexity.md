# AI 说「完成了」时你该问什么：loop 工程的两个陷阱

> 2026-06-11 · 一场两天 Telegram-Claude bridge debug 马拉松的复盘

---

## 背景

我在构建一套「loop 工程」系统：把 Claude Code 接入 Telegram，让我能用手机给 AI 助理下指令、收汇报。想法很酷——从任何地方指挥一台 AI 驱动的 Mac。

实际花了两天、N 次服务重启、最后意识到最大的问题不是技术的。

---

## 陷阱 1：AI 的「完成声明」和「能用」是两件不同的事

### 发生了什么

我让 Claude 搭好这套管道，它每次说「eval 9/9 通过、已发送 Telegram、功能完整」。我以为快好了，下一条指令发出去——没有回应。

这个循环重复了很多次。

### 根因

Claude 的「验证通过」全都是**单元测试**：用模拟输入跑脚本逻辑。没有一次是**端到端真实验证**——用真手机发真消息，等真回复到达。

两者之间有一道真实的鸿沟：
- 单元测试通过 ✅ → 零件没坏
- 端到端通过 ✅ → 整个管道通

在一个有 6 个组件串联的系统里（手机 → Telegram → cloudflare tunnel → webhook server → tmux → Claude → Stop hook → 回 Telegram），单元测试通过率 = 0 的信息量。

### 可固化的规则

**「完成」的定义只有一个：你从真实入口触发，在真实出口收到了预期结果。**

不是「本地 test 过」。不是「Claude 说完成了」。

在 AI 协作里，每次 AI 说「完成/验证过/没问题」，值得问一句：  
**「这是从真实入口到真实出口的端到端验证，还是只测了某个组件？」**

这一个问题能截断 80% 的「怎么还没好」循环。

---

## 陷阱 2：自动化系统的边际产出在某个点会变负

### 发生了什么

我已经有：35 个 launchd 自动化任务、每日晨报、项目体检报告、信号筛选管道……这次又加了双向 Telegram 指挥系统。

调试这个系统用了大量 `claude --print` 调用（几十次 agent spawn、eval loop），把 Anthropic Max 共享配额池压到限流——结果被调试的系统本身（bridge）开始回复要等 14 分钟。

我在用配额测试「快速回复」功能。

### 根因

一人公司 + AI 工具有一个隐形临界点：

**系统越来越多 → 互相争夺有限资源（配额、注意力、端口、内存）→ 边际产出开始为负。**

自动化一开始是解放时间的。过了临界点，它开始消耗时间——维护、调试、重启、互相撞车。

这次还有另一个层面：调试工作本身也是消耗，而且消耗的是被调试对象赖以运作的同一个资源池。

### 可固化的规则

在复杂度已经很高的系统上加新组件之前，值得先问：

1. **这个新功能解决的问题，是真实痛点还是想象中的痛点？** 我每周真的需要从手机实时指挥 AI 几次？
2. **现有系统是否已经覆盖了这个需求的 80%？** 每早一条 Telegram 晨报，已经把状态推过来了。
3. **这个新组件挂掉的时候，修复它需要多少时间？** 串联系统 = 每个组件都是单点故障。

「更多自动化 = 更高效」只在复杂度还低的时候成立。

---

## 两个陷阱的共同根因：节奏问题

回看两天的对话记录，有一个清晰的模式：

```
发现 bug → 立刻改 → 立刻重启 → 新 bug 出来 → 继续改 → …
```

每一步看起来在推进，但在系统层面，这种快速修的节奏制造了累积的脆性：N 次重启、bridge 多次停机、配额耗尽。

更好的节奏是：

```
收集 3 个症状 → 形成完整假设 → 一次性验证 → 再动手
```

「快」和「急」是两件事。快是低摩擦地执行正确的动作；急是在还不确定问题根因的时候快速行动。急往往让一个问题变成三个。

---

## 最终提炼

| 陷阱 | 检测信号 | 中断指令 |
|------|---------|---------|
| AI 完成声明≠能用 | 「测过了 / 验证了 / OK 了」 | 「端到端在真实环境验证了吗？」 |
| 自动化复杂度过载 | 同一系统第 3 次出问题 | 「这个组件的 ROI 是正的吗？不存在会怎样？」 |
| 节奏过急 | 连续 3 次改同一个 bug | 「停。先列完所有症状再动手」 |

这三个中断问题不需要技术背景就能问。它们能让任何 AI 协作者（包括 Claude）慢下来，做出更可靠的动作。
